# Workout App V2 — Design Spec
_Date: 2026-05-11_

## What we're building

A set of targeted upgrades to the live workout app (github.com/ABKodes/workout-app) that make the gym experience seamless — no keyboard, no thinking about what's next — and motivating via visible progress over time. Data is stored offline-first (localStorage) and synced to Supabase in the background when online, tied to a Google account so nothing is ever lost.

---

## Problems being solved

1. **Logging friction** — current text inputs require a keyboard mid-set with sweaty hands
2. **No flow guidance** — user has to manually find the next exercise/set
3. **No motivation loop** — no way to see weights going up or body weight going down over weeks
4. **Data fragility** — localStorage alone is lost if browser is cleared or phone is reset

## Architecture

### Stack additions
| Addition | Purpose |
|---|---|
| `@supabase/supabase-js` | Auth + database client |
| Supabase project (free tier) | PostgreSQL DB + Google OAuth |
| Environment variables | `NEXT_PUBLIC_SUPABASE_URL` + `NEXT_PUBLIC_SUPABASE_ANON_KEY` |

### Sync strategy: offline-first
```
User action (log set, log weight)
  → write to localStorage immediately (instant, always works)
  → if online: upsert to Supabase in background (fire-and-forget)
  → if offline: mark entry as "pending_sync: true"

On app load:
  → if logged in + online: fetch all user rows from Supabase
  → merge with localStorage (Supabase wins on same-date conflict)
  → update localStorage with merged result

On regaining connection (online event):
  → find all localStorage entries with pending_sync: true
  → batch upsert to Supabase
  → clear pending_sync flags
```

### Database schema (Supabase SQL)
```sql
-- Row-level security: users can only access their own rows

CREATE TABLE session_logs (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id      UUID REFERENCES auth.users NOT NULL,
  date         DATE NOT NULL,
  day_index    INTEGER NOT NULL,          -- 0=Mon…6=Sun
  exercises    JSONB NOT NULL DEFAULT '{}',
  session_note TEXT DEFAULT '',
  completed    BOOLEAN DEFAULT FALSE,
  updated_at   TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE(user_id, date, day_index)
);

CREATE TABLE body_weight_entries (
  id        UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id   UUID REFERENCES auth.users NOT NULL,
  date      DATE NOT NULL,
  weight    DECIMAL(5,2) NOT NULL,        -- kg
  UNIQUE(user_id, date)
);

-- RLS policies (both tables follow same pattern)
ALTER TABLE session_logs ENABLE ROW LEVEL SECURITY;
CREATE POLICY "users own their logs"
  ON session_logs FOR ALL
  USING (auth.uid() = user_id);
```

## Feature specs

### Auth + Sync (useAuth.ts + AuthGate.tsx)

**Sign-in flow:**
- App wraps in `<AuthGate>`. If no Supabase session → show `<SignInScreen>`.
- SignInScreen: full dark screen, app name, "Sign in with Google" button (Supabase OAuth), tagline "Your data syncs across any device."
- After sign-in: Supabase stores session token in localStorage, auto-refreshes. User stays logged in permanently unless they sign out.
- Sign-out button: in Stats screen header (small, bottom of screen). Clears Supabase session but keeps localStorage data intact.

**First login migration:**
- On first successful sign-in, check if `wlog_v1` or `bwlog_v1` exist in localStorage with data.
- If yes: batch upsert all entries to Supabase under the user's ID.
- Show "✓ Synced your existing data" toast. Runs once, never again.

**SyncIndicator:**
- Tiny dot (6px) in top-right corner of every screen.
- Grey = offline, green pulse = syncing, hidden = all synced.
- No text, no interruption — just awareness.

**SessionLog type update:**
```typescript
interface SessionLog {
  date: string
  dayIndex: number
  exercises: Record<string, ExerciseLog>
  sessionNote: string
  completed: boolean
  pendingSync: boolean   // ← new: true when not yet uploaded
}
```

**useLog sync behaviour:**
- `logSet`, `setNote`, `finishSession` all set `pendingSync: true` on the session.
- After each write, if `navigator.onLine`, fire `supabase.from('session_logs').upsert(...)` in background (no await in the hot path).
- App listens to `window.addEventListener('online', syncPending)` to flush queue when connection returns.

### 1. Stepper Logging (SetRow rewrite)

**Replaces:** two text `<input>` fields for weight and reps.

**New UI per set:**
- "Same as last time" pill button — visible only when prevLog exists. One tap fills weight + reps from last session and marks set done + starts timer.
- Weight stepper: `−` | `55 kg` | `+` — step size 2.5 kg. Pre-loaded from prevLog.
- Reps stepper: `−` | `12` | `+` — step size 1. Pre-loaded from prevLog.
- Done ✓ button — unchanged.

**Behaviour:**
- Values start at prevLog values. If no prevLog exists, weight defaults to 20 (bar weight) and reps defaults to the lower bound of the exercise's reps range.
- No keyboard ever appears.
- Weight decrements minimum 0, reps minimum 1.

---

### 2. Guided Session Mode (GuidedSession.tsx)

**Entry:** "🎯 Start Guided Session" button at top of DayScreen for gym days only. Full-screen fixed overlay (z-60).

**State:** `currentExIdx` (0…n exercises) + `currentSetIdx` (0…n sets).

**Screen layout (top to bottom):**
1. Top bar: progress bar (% of total sets done), `✕` exit button top-right
2. Exercise context: "Exercise 3 of 6" label, exercise name (large), note (small grey)
3. Set pills: all sets for this exercise (done = green, current = orange, upcoming = grey)
4. Big steppers: weight (step 2.5kg) and reps (step 1), pre-loaded from prevLog
5. "Same as last time" pill if prevLog exists
6. Primary button: "✓ Done — start rest timer"
7. "What's next" preview: "After rest: Set 2 of Lat Pulldown"

**Flow:**
- Tap done → logs set → starts RestTimer → auto-advances when timer finishes
- All exercises done → "Session complete 🔥" screen: total sets, PRs hit, session note input, Finish button, auto-progression tips
- ✕ exits to normal DayScreen (data already saved)

### 3. Body Weight Tracker (BodyWeightSection.tsx + useBodyWeight.ts)

**Data model:**
```typescript
interface BodyWeightEntry {
  date: string    // "2026-05-11"
  weight: number  // kg
  pendingSync: boolean
}
// localStorage: "bwlog_v1" → BodyWeightEntry[]
// Supabase: body_weight_entries table
```

**Screen layout:**
1. Three stat cards: Today's weight · Total lost (kg) · Rate kg/week (rolling 2-week average)
2. SVG line chart: last 5 weeks. Actual weight (green) + target trend line (orange dashed, −0.6kg/week from start weight)
3. "Log today's weight" → inline expand with number input + confirm. One entry per day (overwrites same-day).
4. Recent entries list: date · weight · delta (green = loss, red = gain)

**Rate calculation:** `(earliestWeight − latestWeight) / weeksElapsed`. Shown in purple.

**Sync:** same offline-first pattern as session logs. `useBodyWeight` mirrors `useLog`'s pendingSync + online event listener.

---

### 4. Strength Progress Charts (ProgressSection.tsx + useProgress.ts)

**Screen layout:**
1. Scrollable exercise picker pills (gym-day exercises only)
2. PR badge: "🏆 Best: 62.5kg × 10" + date
3. SVG line chart: heaviest set weight per session, orange line + gradient fill, star on PR session
4. Session history bars: horizontal bars per session, darker = older. Date + weight label.

**Data derivation:**
- Read all SessionLog entries for the exercise name
- Extract max weight per session from done sets
- Sort ascending by date
- PR = global max weight

**Exercise picker default:** based on currently active day tab.

---

### 5. PR Detection + Celebration (PrCelebration.tsx)

**Trigger:** in DayScreen and GuidedSession, after `logSet`, compare new weight against `useProgress(exerciseName).prWeight`. If exceeded, set `newPR` state.

**Overlay:** full-screen dark overlay. Centre: 🏆 + "New PR!" + weight. CSS burst animation (scale + opacity, no library). Auto-dismisses after 2.5s or on tap.

---

### 6. Auto-progression Tip

**Trigger:** Session complete screen (guided + normal finish).

**Logic:** parse upper bound from `exercise.reps` (last integer in string: "10–12" → 12). If all logged sets have reps ≥ upper bound, suggest +2.5kg.

**UI:** "💡 Lat Pulldown — you hit all reps. Try 62.5kg next session."

---

### 7. Warm-up Sets Calculator (WarmupCalc.tsx)

**Entry:** "🔥 warm-up" button in ExerciseCard button row. Bottom-sheet modal.

**Input:** working weight (pre-filled from last session log).

**Output:**
| Set | % | Weight | Reps |
|---|---|---|---|
| 1 | 50% | 27.5 kg | 8 |
| 2 | 70% | 37.5 kg | 5 |
| 3 | 85% | 47.5 kg | 3 |

Weights rounded to nearest 2.5kg. Gym-day exercises only.

## Data safety

- **Offline:** localStorage is always written first. App works fully without internet.
- **Online:** Supabase syncs in background. Phone can be reset or switched — data loads from cloud on next login.
- **Conflict resolution:** Supabase wins on same-date conflict during merge on load. Local pending writes win over stale cloud data during flush.
- **Export button** (in Stats screen): still included as manual backup. Downloads `wlog_v1` + `bwlog_v1` as JSON.
